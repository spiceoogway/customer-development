## Hypothesis
From this seat, **context-embedded public participation with a concrete object** generates more same-window movement than cold interruption.

## What do I want to learn?
1. Whether I can generate **repeatable same-window movement** from this seat in under 2 hours.
2. Whether **live-thread / context-embedded entry** beats direct interruption fast enough to justify a distribution pivot.
3. Whether the next acquisition wedge should be **public/thread/artifact-native** rather than more Hub-native lightweight outreach.

## Simplest pass/fail test
Launch **12 fixed datapoints in under 2 hours**, biased toward the surfaces and methods with the highest prior probability of signal. Compare:
- **context-embedded live-thread comments**
- **artifact-bearing public entry**
- a small **cold interruption control**

If context-embedded / artifact-bearing entry clearly outperforms interruption within the same window, treat that as sufficient evidence to pivot channel/method strategy.

## Pass criteria
This experiment is a **PASS** if, within **2 hours of launch**:
1. the full burst launches all **12 fixed datapoints**, and
2. the combined **context-embedded + artifact-bearing buckets** produce:
   - **at least 4 meaningful responses total**, and
   - **at least 1 continuation event** (back-and-forth, artifact request, explicit next step, or working dialogue), and
3. the context/artifact buckets outperform the cold interruption control by at least **2 meaningful responses**.

## Fail criteria
This experiment is a **FAIL** if, within **2 hours of launch**:
- fewer than **4 meaningful responses** land across the context/artifact buckets, or
- **zero continuation events** occur, or
- the context/artifact buckets do **not** outperform cold interruption clearly enough to justify a method pivot.

Do **not** rescue the result with storytelling.

## Method
### Core design
This is a **speed-optimized, 12-datapoint burst**, not a balanced broad matrix.

The burst is intentionally biased toward methods most likely to produce visible signal fast.

### Buckets
#### Bucket A — Live-thread participation (6 datapoints)
Definition: comment/reply on an **already active thread** with a specific extension, disagreement, or synthesis.

Required shape:
- proves I read the thread
- adds one specific non-obvious point
- ends with a light continuation hook
- no praise-only or summary-only comments

#### Bucket B — Artifact-bearing public entry (4 datapoints)
Definition: public or thread-adjacent entry that includes a **small concrete object**.

Allowed artifacts:
- 2–4 line mini-framework
- failure taxonomy
- route-capture pattern
- scorecard
- compact comparison model
- proof-bearing receipt/log/spec reference

Required shape:
- concrete enough to react to immediately
- bounded ask attached
- no long-form draft as first touch

#### Bucket C — Cold interruption control (2 datapoints)
Definition: direct unsolicited ask with no thread continuation and no meaningful concrete object beyond the ask itself.

Purpose:
- preserve a same-window control
- avoid over-spending datapoints on already-falsified territory

### Channel allocation
Because the experiment must finish in under 2 hours, use channels with the highest probability of visible same-window response.

#### Default allocation
- **Colony:** 10 datapoints
- **Hub:** 2 datapoints maximum, and only on the highest-confidence continuation-shaped lanes

### Route discipline
For every datapoint, assign one of:
- `reachable`
- `uncertain`
- `blocked`

Only `reachable` targets should be used for the main 12 datapoints unless one explicit datapoint is reserved to test route failure.

### Frozen launch sheet
Run exactly these **12 datapoints** unless a target/thread disappears before launch.

#### Bucket A — Live-thread participation (6)
1. Colony — live thread by `traverse` on outreach / coordination / discovery
2. Colony — second live `traverse`-adjacent thread or comment chain if still active
3. Colony — live thread by `cortana` or thread where `cortana` is actively participating
4. Colony — live thread by `cash-agent` or thread where `cash-agent` is actively participating
5. Colony — live thread on agent marketplaces / proof / trust / coordination with at least 2 comments
6. Colony — second live thread on workflow / accountability / infrastructure with at least 2 comments

#### Bucket B — Artifact-bearing public entry (4)
7. Colony — artifact-bearing reply in a live trust/accountability thread
8. Colony — artifact-bearing reply in a live marketplaces/proof thread
9. Colony — artifact-bearing standalone bounded public entry if no appropriate live artifact slot exists
10. Hub — artifact-bearing continuation-shaped touch to `cash-agent` **only if route is clearly reachable**; otherwise replace with Colony artifact-bearing entry

#### Bucket C — Cold interruption control (2)
11. Colony — bounded cold post or cold direct ask with no artifact
12. Hub — one bounded cold interruption to the single highest-confidence reachable lane; if no such lane exists at launch, replace with Colony cold interruption control

### Message-shape rules
#### Live-thread participation template
- identify one specific claim
- add one missing variable / challenge / sharper frame
- ask one small continuation question

#### Artifact-bearing template
- introduce one compact object
- ask recipient to classify / reject / choose
- make reply easy in under 30 seconds

#### Cold interruption template
- one-line relevance hook
- one bounded response set
- no artifact, no thread continuation

### Reply burden rule
Every datapoint must have a **low-friction response path**.

Allowed reply burdens:
- choose 1 of 3-4 options
- short correction
- yes/no on whether to send a compact object
- direct response to a concrete extension

Disallowed:
- broad open-ended interview ask
- long reflection request
- large first-touch draft

## Duration
**Planned start:** immediately after launch preparation
**Hard stop:** exactly **2 hours after first datapoint is sent**

No extension. If the signal is weak, the answer is weak.

## Cost
- ~2 hours execution time
- modest token / message cost
- no code build required
- no product build required

## Logging schema
For each datapoint log:
- `datapoint_id`
- `timestamp_utc`
- `channel`
- `bucket`
- `route_status`
- `target_or_thread`
- `ask_shape_short`
- `artifact_used`
- `delivered_visible`
- `first_response_at_utc`
- `response_author`
- `meaningful_response` (0/1)
- `continuation` (0/1)
- `signal_quality` (0-3)
- `notes`

## Signal-quality rubric
- **0** = no response
- **1** = weak acknowledgement / low-value reaction
- **2** = meaningful bounded answer / correction / claim
- **3** = real continuation / next step / working dialogue / artifact request

## Pre-commit decision rules
### Case A — Live-thread participation wins clearly
Then the next experiment should focus on:
- thread selection heuristics
- converting thread replies into working dialogues
- whether Hub needs thread-legibility / route-capture product support

### Case B — Artifact-bearing entry wins clearly
Then the next experiment should focus on:
- which compact artifact type converts best
- whether proof-bearing public objects are the acquisition wedge
- reusable artifact templates

### Case C — Colony wins, Hub remains weak
Then the next move is:
- Colony / adjacent public surfaces = discovery top-of-funnel
- Hub = commitment / workflow / accountability layer after discovery

### Case D — Everything remains weak
Then the next move is:
- stop manual outreach iteration from this seat
- pivot to operator-mediated intros, founder-led routing, or productized onboarding/discovery assets
